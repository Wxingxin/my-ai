这个文件是**系统提示词管理模块**，把传给大模型的 system prompt 拆分成可组合的片段，按场景灵活拼装。

---

**三个提示词片段**

`BASE` 是所有场景都需要的基础信息，注入了 AI 身份、当前日期、语言和格式要求。日期用 `new Date().toLocaleDateString("zh-CN")` 动态生成，每次模块加载时自动更新，不需要手动维护。

`TOOLS_GUIDE` 告诉模型什么情境下调工具，以及"不要输出原始 JSON"这条规则非常关键——没有它，模型很可能直接把 `{ temp: 22, humidity: 60 }` 这样的原始数据返回给用户。

`BEHAVIOR` 定义调用策略，其中"意图明确时直接调用，不征求同意"避免了模型每次都问"需要我帮你查一下吗？"这种多余确认，提升使用体验。

---

**三个导出的完整提示词**

| 导出            | 构成                          | 适用场景      |
| --------------- | ----------------------------- | ------------- |
| `SYSTEM_PROMPT` | BASE + TOOLS_GUIDE + BEHAVIOR | 通用入口      |
| `CHAT_PROMPT`   | BASE + TOOLS_GUIDE + BEHAVIOR | 聊天页面      |
| `TODO_PROMPT`   | BASE + "专注待办"             | Todo 专用页面 |

`SYSTEM_PROMPT` 和 `CHAT_PROMPT` 当前内容完全一样，这是为了**以后分化预留的扩展点**，比如聊天场景将来可能需要加情感风格，而通用入口保持中性，提前拆开比到时候再改要安全得多。

`TODO_PROMPT` 去掉了 `TOOLS_GUIDE` 和 `BEHAVIOR`，说明 Todo 专用页面可能不需要天气、新闻这些工具，通过提示词层面就做了功能收敛，而不是改代码逻辑。

---

**整体设计思路**

用字符串拼接代替单一大段提示词，让每个片段职责单一、独立可维护。新增场景时只需组合已有片段，不用每次从头写完整 prompt，这是提示词工程里的**模块化**实践。
