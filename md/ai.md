# AI 是怎么知道要调用 weather、news、todo 这些能力的

这份文档解释两个问题：

1. AI 为什么“知道”你的话是在让它查天气、查新闻、操作 Todo。
2. 这种能力在工程上通常是怎么实现的，有哪些常见做法。

这不是 AI 天生就会“自动调用你本地函数”。
真正发生的事情是：

- 你先把“它能调用什么工具”告诉模型。
- 你再把“什么时候应该调用这些工具”告诉模型。
- 模型根据用户输入做意图判断。
- 如果判断应该调用工具，它会返回一个结构化的“我要调用哪个工具、参数是什么”。
- 服务端代码收到后，真的去执行你的 weather/news/todo 逻辑。
- 然后再把工具结果发回给模型，让它整理成自然语言回复给用户。

所以本质上不是“AI 自己直接调用 API”，而是：

`用户自然语言 -> 模型判断意图 -> 模型产出 tool call -> 服务端执行工具/API -> 结果回注给模型 -> 模型组织最终答案`

---

## 一、先说结论：AI 为什么知道你在让它调用工具

因为你提前给了它三样东西：

1. 工具列表
2. 工具说明
3. 调用规则

在这个项目里，这三样东西都很明确。

### 1. 工具列表：告诉 AI “你可以做什么”

比如项目里定义了这些工具：

- `get_weather`
- `get_news`
- `get_todos`
- `add_todo`
- `complete_todo`

对应代码可以看这里：

- `\my-ai\lib\tools\index.ts`
- `\my-ai\lib\tools\weather.ts`
- `\my-ai\lib\tools\news.ts`
- `\my-ai\lib\tools\todos.ts`

这些工具不是写给用户看的，而是写给模型看的。
模型会看到每个工具的：

- 名字
- 功能描述
- 参数结构

例如：

- `get_weather(city)`
- `get_news(keyword)`
- `add_todo(title, description, priority, category, dueDate)`

这一步很重要。
如果你根本没有把工具定义传给模型，模型就不可能返回合法的工具调用。

### 2. 工具说明：告诉 AI “这个工具是干嘛的”

例如 `get_news` 的描述是“获取最新新闻，可按关键词搜索；如果没有关键词，则返回当前头条新闻”。

这类说明本质上是在帮模型建立“语义映射”：

- 用户说“最近 AI 行业有什么热点”
- 模型会联想到：这和“新闻”“热点”“资讯”有关
- 它再去匹配哪个工具最合适
- 最后选择 `get_news`

也就是说，模型不是靠写死的 `if user contains 新闻 then get_news` 才能工作。
它可以利用语言理解能力，把用户表达和工具描述做语义对齐。

### 3. 调用规则：告诉 AI “什么时候必须用工具”

在这个项目中，最关键的是系统提示词里的规则。

文件：

- `\my-ai\lib\prompts\system.ts`

里面明确写了类似规则：

- 用户问天气、气温、气候 -> 必须调用 `get_weather`
- 用户提到任务/待办/todo -> 调用 `get_todos` / `add_todo` / `complete_todo`
- 用户问新闻/资讯/热点 -> 调用 `get_news`

这就相当于告诉模型：

- 不是所有问题都直接口头回答
- 遇到某些意图时必须先调工具

所以模型之所以“知道”，不是神秘能力，而是因为：

- 它看到了工具定义
- 它看到了工具用途
- 它看到了系统规则
- 它本身有自然语言理解能力

于是它就能把“帮我看看今天北京天气”理解成“该调用 `get_weather`，参数 city=北京”。

## 二、这个项目里实际是怎么跑起来的

下面按真实链路讲。

---

## 三、完整调用链路

## 第 1 步：用户输入自然语言

例如用户说：

- “北京今天天气怎么样”
- “帮我新增一个 Todo：晚上写简历”
- “最近 AI 有什么新闻”

前端只是把这句话发给后端，并不会自己决定调用哪个 API。

例如聊天接口：

- `\my-ai\app\api\chat\route.ts`

Todo AI 接口：

- `\my-ai\app\api\todos\ai\route.ts`

新闻接口：

- `\my-ai\app\api\news\route.ts`

---

## 第 2 步：后端把“消息 + 工具 + 系统提示词”一起发给模型

在聊天场景里，后端会组装一份 `fullMessages`：

- `system`：系统提示词，规定行为
- 历史对话：让模型理解上下文
- 当前用户输入：本轮问题

然后把工具定义也传给模型。

核心代码：

- `\my-ai\app\api\chat\chat-service.ts`
- `\my-ai\lib\models\index.ts`

你可以把它理解成：

“这是用户的话，这是历史上下文，这些是你可用的工具，请你决定是直接回答，还是调用工具。”

---

## 第 3 步：模型做“意图识别 + 工具选择”

这一步是关键。

模型内部会做类似这样的判断：

### 例子 1：天气

用户说：

`上海明天会下雨吗`

模型会理解出：

- 领域：天气
- 动作：查询
- 参数：城市 = 上海

于是它会返回一个工具调用，而不是直接瞎回答。

类似：

```json
{
  "tool_calls": [
    {
      "id": "call_1",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\":\"上海\"}"
      }
    }
  ]
}
```

### 例子 2：Todo

用户说：

`帮我新增一个待办，明天上午整理项目经历`

模型会理解出：

- 领域：Todo
- 动作：新增
- 标题：整理项目经历
- 时间信息：明天上午

如果工具参数里支持这些字段，它就会返回 `add_todo(...)`。

### 例子 3：新闻

用户说：

`最近 AI 行业有什么热点`

模型会理解出：

- 领域：新闻
- 主题：AI

于是它会返回 `get_news(keyword="AI")`。

所以这里的“知道”其实就是：

- 大模型做意图识别
- 再基于工具 schema 生成结构化调用

---

## 第 4 步：服务端真正执行工具

注意，模型只是在“建议调用什么”。
真正执行的是你的服务端代码，不是模型本身。

例如这里：

- `\my-ai\lib\tools\index.ts`
- `\my-ai\app\api\chat\chat-service.ts`

逻辑是：

1. 解析模型返回的 `tool_calls`
2. 读取工具名和参数
3. 在服务端匹配到具体函数
4. 调用真实逻辑

例如：

- `get_weather` -> 调天气 API
- `get_news` -> 拉 RSS / News 数据
- `add_todo` -> 写数据库
- `complete_todo` -> 更新数据库

也就是说，AI 从来没有真正接触你的数据库或第三方 API 凭证。
它只是输出：

“我建议你调用这个工具，参数是这些。”

真正动手的是你的后端。

---

## 第 5 步：把工具结果再发回给模型

工具执行完以后，后端不会直接把原始 JSON 扔给用户。
而是会把工具结果作为一条 `tool` 消息，再塞回模型上下文。

例如：

```json
{
  "role": "tool",
  "tool_call_id": "call_1",
  "content": "{\"temperature\":18,\"condition\":\"晴\"}"
}
```

然后再让模型继续回答。

这一步的作用是：

- 工具负责拿真实数据
- 模型负责把数据组织成人类易读的话

所以最后用户看到的是：

`北京今天晴，气温 18°C，体感较舒适，适合出行。`

而不是：

```json
{ "temperature": 18, "condition": "晴" }
```

---

## 第 6 步：如果有需要，会继续多轮工具调用

这个项目里聊天链路支持多轮工具调用，不是只能调一次。

代码里有：

- 最大轮次限制 `MAX_TOOL_ROUNDS = 6`

文件：

- `\my-ai\app\api\chat\chat-service.ts`
- `\my-ai\app\api\todos\ai\route.ts`

这代表模型可以这样工作：

1. 先调用 `get_todos`
2. 看到现有 Todo 列表后，再决定调用 `complete_todo`
3. 最后再给用户一个总结

这已经接近 Agent 的执行模式了。

## 四、这个项目里一共有几种“AI 知道要干什么”的实现方式

这个仓库里其实不止一种。

---

## 方式 1：标准 Function Calling / Tool Calling

这是聊天主链路最标准、最推荐的一种。

特点：

- 把工具定义传给模型
- 模型返回结构化调用
- 服务端执行
- 结果回注模型

项目里的典型位置：

- `\my-ai\app\api\chat\chat-service.ts`
- `\my-ai\lib\tools\index.ts`

优点：

- 扩展性强
- 结构清晰
- 不容易写死
- 模型能自己做选择

缺点：

- 依赖模型对工具调用能力的支持
- 参数 schema 写不好时，模型可能乱传参
- 需要做好轮次控制、异常处理、鉴权

适用场景：

- 通用 AI 助手
- 多工具系统
- 复杂业务操作

---

## 方式 2：Prompt 规则驱动

这个项目里也大量用了这种方式。

例如在系统提示词里直接写：

- 问天气必须调用天气工具
- 提到 Todo 就调用 Todo 工具
- 问新闻就调用新闻工具

文件：

- `\my-ai\lib\prompts\system.ts`

这本质上是在做“软路由”。

优点：

- 简单
- 成本低
- 适合快速迭代

缺点：

- 提示词不稳定时，模型可能不听话
- 规则复杂后会越来越难维护

适用场景：

- 规则较少
- 工具种类不多
- 需要快速验证产品

---

## 方式 3：代码里先做人为路由，再决定调哪个工具

这种方式的思路是：

- 不完全依赖模型判断
- 先在代码里做一层意图分类
- 分类完再调用对应 API 或工具

例如可以先用规则判断：

- 包含“天气、下雨、温度” -> weather
- 包含“新闻、热点、资讯” -> news
- 包含“待办、todo、任务” -> todo

然后只把对应工具传给模型，或者直接执行对应后端逻辑。

这个仓库里的新闻模块其实有点接近这种思路。

文件：

- `\my-ai\app\api\news\route.ts`

这里做了一个 `extractKeyword()`：

- 先从用户话里抽关键词
- 再直接调用 `getNews(keyword)`
- 再把结果交给模型总结

也就是说，新闻模块不是“完全让模型自己决定要不要调工具”，而是后端先帮它做了部分路由和参数提取。

优点：

- 可控性强
- 更稳定
- 对简单业务很有效

缺点：

- 规则多了以后会变硬编码
- 泛化能力弱

适用场景：

- 垂直场景
- 问题类型比较固定
- 对稳定性要求高

---

## 方式 4：让模型只做“分类器”

还有一种常见做法是：

- 先让一个模型只输出意图分类结果
- 再由你的代码决定调用哪个系统

例如先让模型返回：

```json
{
  "intent": "todo_create",
  "confidence": 0.92,
  "entities": {
    "title": "整理项目经历"
  }
}
```

然后由后端执行：

- `intent=todo_create` -> `add_todo`
- `intent=weather_query` -> `get_weather`

这和 function calling 的区别是：

- function calling：模型直接产出“调用哪个函数”
- classification：模型先产出“这是什么意图”，函数调用由代码决定

优点：

- 更可控
- 更适合复杂审批流
- 更容易记录和审计

缺点：

- 比 function calling 多一层
- 开发量更大

适用场景：

- 金融、医疗、企业流程
- 高风险动作
- 需要人工审核的系统

---

## 方式 5：工作流 / Agent 编排

再往上走，就是 Agent 或工作流系统。

例如：

1. 模型判断用户想完成什么任务
2. Planner 拆成多个步骤
3. 执行器逐步调用不同工具
4. 每步结果再回传给模型
5. 最后输出完整结论

这个项目的聊天链路已经有一点这种味道，因为它支持：

- 多轮工具调用
- 工具结果回注
- 最终总结回答

只是还没有拆成显式的 Planner / Executor / Memory 模块。

适用场景：

- 复杂任务执行
- 跨多个系统联动
- 长链路任务

## 六、AI 是怎么“从一句中文里理解出参数”的

这是很多人最容易疑惑的地方。

例如：

`帮我看看杭州今天会不会下雨`

模型为什么知道参数是：

```json
{ "city": "杭州" }
```

原因是：

1. 模型本身具备语言理解能力
2. 你给了它参数 schema
3. 你给了它工具说明

比如工具 schema 告诉它：

- `get_weather` 需要一个 `city`

模型就会尝试从用户输入里抽取“最像城市名的实体”。

再比如：

`帮我新增一个高优先级工作任务：明天提交简历`

模型可能解析成：

```json
{
  "title": "明天提交简历",
  "priority": "high",
  "category": "work"
}
```

这本质上就是：

- 意图识别
- 实体抽取
- 参数结构化

这也是为什么工具参数定义越清晰，模型表现通常越稳定。

## 七、为什么有时候 AI 会判断错

因为它不是“精确规则引擎”，而是“概率模型”。

常见问题有：

### 1. 工具描述不清楚

如果两个工具描述很像，模型会选错。

例如：

- `search_info`
- `get_news`

如果描述都写得很模糊，模型就可能乱选。

### 2. 提示词约束不够强

如果你只说“可以使用工具”，模型可能直接口头回答，不调工具。

如果你明确写“用户问天气必须调用 get_weather”，效果通常更稳。

### 3. 参数 schema 设计太弱

例如工具需要 `city`，但你没写清楚字段意义，模型可能传出：

- `location`
- `address`
- `place`

而不是你需要的 `city`

### 4. 用户意图本身含糊

例如：

`帮我安排一下`

这句话到底是：

- 新增 Todo
- 查看现有 Todo
- 给建议

其实并不明确。

这时更好的做法不是乱调用，而是先追问。

### 5. 工具返回结果不够友好

如果工具返回的是极乱的 JSON，模型总结效果也会变差。

所以工具层最好先做一次清洗和结构化。

## 八、工程上怎么把这个能力做稳

如果你以后继续做这类系统，建议注意下面几点。

### 1. 工具定义一定要清晰

一个好的工具定义要写清楚：

- 这个工具解决什么问题
- 参数是什么意思
- 参数有哪些约束
- 什么时候该用它
- 什么时候不该用它

### 2. 工具不要设计得太泛

差的设计：

- `do_task(anything)`

好的设计：

- `get_weather(city)`
- `get_news(keyword)`
- `add_todo(title, priority, category)`

工具越明确，模型越容易选对。

### 3. 高风险操作要加二次确认

例如：

- 删除 Todo
- 修改密码
- 删除文件
- 发邮件

虽然模型可以调用工具，但不代表所有动作都该“直接执行”。

可以采用：

- 先让模型生成操作意图
- 前端弹确认框
- 用户确认后再真正执行

### 4. 给工具调用设置最大轮次

这个项目里已经做了：

- `MAX_TOOL_ROUNDS = 6`

否则模型可能一直：

- 查一下
- 再查一下
- 再补一步

最后死循环。

### 5. 工具执行一定要做权限校验

例如 Todo 工具必须绑定当前用户：

- 只能查自己的 Todo
- 只能改自己的 Todo

不能因为模型说“帮我完成 id=xxx”就直接修改任意数据。

真正的安全边界一定在后端，不在模型。

### 6. 模型只负责建议，代码负责最终执行

永远不要把“是否执行危险操作”的决定权完全交给模型。

正确原则是：

- 模型负责理解和建议
- 代码负责校验和执行

## 九、如果不用 Function Calling，还有哪些实现方案

总结一下常见方案。

### 方案 A：纯提示词 + 纯文本输出

做法：

- 让模型自己回答
- 不接真实工具

优点：

- 最简单

缺点：

- 不能访问实时数据
- 容易幻觉
- 不能真正执行操作

适用：

- 纯内容生成

### 方案 B：规则路由

做法：

- 代码里写关键词规则
- 命中后直接调用后端 API

优点：

- 稳定
- 好控

缺点：

- 不灵活
- 难扩展

适用：

- 简单垂直场景

### 方案 C：分类器 + 执行器

做法：

- 模型先输出意图
- 后端根据意图执行

优点：

- 可控
- 好审计

缺点：

- 比 function calling 更复杂

适用：

- 企业级流程

### 方案 D：Function Calling

做法：

- 把工具直接暴露给模型
- 模型返回结构化调用
- 服务端执行

优点：

- 最自然
- 最适合通用 AI 助手

缺点：

- 对工具 schema 和提示词要求高

适用：

- 多工具 AI 产品

- 执行
- 反思
- 多轮工具协作

优点：

## 十、你这个项目最核心的方法是什么

如果一句话总结：

这个项目最核心的方法是：

**用系统提示词约束模型行为，用 function calling 让模型输出结构化工具调用，再由服务端安全地执行真实 API / 数据库操作，最后把结果回注给模型生成自然语言回复。**

拆开看是四层：

1. Prompt 层：规定什么时候该调工具
2. Tool Schema 层：定义有哪些工具、参数是什么
3. Orchestrator 层：解析 `tool_calls`，执行工具，控制轮次
4. Tool/Business 层：真正访问 weather/news/todo/DB/API

这四层缺一不可。

## 十一、你可以怎么向别人解释这套机制

你可以这样说：

“AI 并不是自己真的会操作天气 API 或数据库，而是我先把可用工具和调用规则告诉模型。模型收到用户输入后，会根据语义判断当前意图，返回结构化的工具调用请求。后端再根据这个请求执行真实的 API 或数据库操作，并把结果回传给模型，让模型整理成用户可读的最终回复。这样既保留了自然语言交互能力，又保证了系统可控、安全和可扩展。”

## 十二、如果后面要升级，这几个方向最有价值

### 1. 更细的工具描述

把每个工具的适用边界写得更清楚，减少误判。

### 2. 参数校验更严格

例如：

- 城市名校验
- Todo 标题长度校验
- 日期标准化

### 3. 高风险动作确认机制

例如删除、覆盖、发送类操作先确认。

### 4. 增加意图分类层

在复杂业务里，先分类再调用工具会更稳。

### 5. 加入日志与可观测性

记录：

- 用户输入
- 模型选择了什么工具
- 参数是什么
- 工具执行结果
- 最终回复是什么

这样排查问题会快很多。

## 十三、最后的核心理解

AI 能“知道你要调用 weather、news、todo”，并不是因为它真的理解了你的系统内部代码。

而是因为你给了它一套“可行动的语义接口”：

- 工具名
- 工具描述
- 参数 schema
- 系统规则
- 历史上下文

模型用语言理解能力把用户请求映射成工具调用；
代码再把这个调用真正落地执行。

所以这套系统本质上不是“AI 直接操作世界”，而是：

**AI 负责理解意图，代码负责执行世界。**
