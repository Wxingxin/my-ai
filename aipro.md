# Aria AI 项目升级与优化方向

这份文档的目标不是泛泛而谈“还能加什么功能”，而是基于你当前这个项目的现状，系统性分析：

1. 现在已经做到了什么
2. 还能往哪些方向升级
3. 哪些地方最值得优先优化
4. 如果你要把项目做成更像正式产品，下一步应该怎么推进

---

## 一、先看当前项目处于什么阶段

你现在这个项目已经不是简单的 AI Demo 了，它已经具备了一个 AI 产品的雏形：

- 有登录注册和基础鉴权
- 有聊天会话管理
- 有多模型接入
- 有工具调用能力
- 有 Todo 管理
- 有新闻检索与 AI 总结
- 有忘记密码邮件流程
- 有 MongoDB 持久化

这说明你已经完成了“从 0 到 1”的阶段。

但如果要继续升级，这个项目接下来要解决的问题，不再只是“能不能跑”，而是：

- 能不能更稳定
- 能不能更像正式产品
- 能不能扩展更多业务能力
- 能不能让 AI 更可靠
- 能不能更方便后续维护和迭代

所以升级方向建议分成 5 个层次：

1. 产品能力升级
2. AI 能力升级
3. 工程架构升级
4. 安全与稳定性升级
5. 数据与运营能力升级

---

## 二、最值得优先升级的方向

如果只说最重要的几个，我认为优先级最高的是下面这些：

### 1. 把 AI 从“能用”升级到“稳定可控”

你现在已经有 tool calling，但还可以继续强化：

- 增加更严格的参数校验
- 增加工具调用日志
- 增加失败重试
- 增加高风险操作确认
- 增加工具调用观测面板

原因很简单：

AI 系统早期最容易出的问题不是“不会回答”，而是：

- 调错工具
- 参数传错
- 调用链路中断
- 工具成功了但用户看不出来

这部分升级后，系统质量会明显上一个台阶。

### 2. 把 Todo 从基础 CRUD 升级成真正的任务系统

现在 Todo 已经能创建、完成、删除，但还不够“生产级”。

可以升级成：

- 支持编辑 Todo
- 支持截止时间
- 支持排序和拖拽
- 支持搜索和标签
- 支持批量操作
- 支持 AI 自动拆解任务

这是最容易做出产品感的模块。

### 3. 聊天模块从“单轮工具调用”升级成“更完整的工作助手”

聊天目前已经支持会话和工具，但还可以进一步增强：

- 对话搜索
- 会话重命名
- Pin 置顶会话
- 上下文压缩
- 长对话摘要
- 聊天内容导出
- 多模态输入

如果这些补上，聊天模块会从“像一个课程作业”变成“像一个可持续使用的产品”。

### 4. 补齐用户资料、设置、文件能力

目前 `profile` 和 `setting` 的后端能力基本还没落地。

这是一个明显的可扩展点。

可以做：

- 用户头像上传
- 用户昵称和简介编辑
- 模型偏好设置
- 默认语言设置
- 默认 provider 设置
- 文件上传后让 AI 读取文件

这会让项目更完整。

## 三、产品功能层面的升级方向

---

## 1. Chat 模块升级

聊天是你项目的核心，建议重点升级。

### 可以扩展的能力

- 会话重命名
- 会话归档 / 删除恢复
- 会话搜索
- 置顶重要会话
- 按时间分组显示会话
- 导出会话为 Markdown / PDF
- 复制 AI 回复
- 消息重试
- 重新生成回答
- 中断流式输出

### 更进一步的升级

- 支持上传文件到对话中
- 支持图片理解
- 支持引用历史消息
- 支持一条消息中多个工具协同
- 支持“请基于我之前的 Todo 和新闻做今天计划”

### 为什么值得做

因为聊天是用户停留时间最长的地方。
如果聊天体验更完整，整个产品价值会被放大。

### 当前存在的优化点

- 当前聊天支持创建和删除对话，但没有重命名
- 没有消息级别操作，比如复制、重试、继续生成
- 没有“正在调用哪个工具”的中间状态展示
- 没有 token 上限或长对话压缩策略
- 没有异常恢复机制，比如流式中断后的补偿

---

## 2. Todo 模块升级

Todo 是最适合做深做实的模块。

### 现在已经有的

- 新增
- 完成
- 删除
- 筛选
- AI 创建 / 完成 / 删除

### 建议补齐的能力

- 编辑 Todo 标题与描述
- 设置和修改截止时间
- 支持 `in_progress` 状态的完整流转
- 优先级排序
- 自定义标签
- 搜索 Todo
- 按分类统计
- 批量完成 / 批量删除
- 历史归档

### 可以做成更强的 AI 能力

- 自动拆解大任务
- 自动生成每日计划
- 自动根据截止时间排序
- 自动识别“高优先级但未完成”的任务
- 自动总结今日已完成/未完成事项
- 聊天里直接引用 Todo 上下文

### 更像产品的升级

- 看板视图
- 日历视图
- 周计划视图
- 任务提醒
- 邮件提醒 / 站内通知

如果这些做出来，这个模块会从“AI Todo 演示”升级为“AI 任务管理系统”。

---

## 3. News 模块升级

新闻模块已经有基本能力，但还比较轻。

### 可以扩展的方向

- 支持订阅主题
- 支持收藏新闻
- 支持稍后阅读
- 支持多来源切换
- 支持按分类过滤
- 支持生成日报 / 周报
- 支持新闻去重优化
- 支持来源可信度排序

### AI 方向的增强

- 同一主题多篇新闻聚合总结
- 自动提炼“发生了什么 / 为什么重要 / 有何影响”
- 根据用户偏好生成个性化新闻流
- 新闻和 Todo 联动

例如：

- 用户看到“某 AI 产品发布”
- 直接生成“调研这个产品”的 Todo

### 当前优化点

- 目前新闻主要依赖 RSS，来源数量还比较少
- 关键词抽取规则比较简单
- 缺少用户偏好和收藏机制
- 缺少新闻历史缓存

---

## 4. Profile / Setting 模块升级

这两个模块现在明显还没做完，是最直接的提升空间。

### Profile 可以做

- 上传头像
- 编辑昵称
- 编辑简介
- 展示注册时间
- 展示最近活跃时间
- 展示消息数 / Todo 数统计
- 展示偏好模型

### Setting 可以做

- 默认模型选择
- 默认语言设置
- 深色 / 浅色主题切换
- 是否开启流式输出
- 是否允许自动调用工具
- 是否允许删除类操作自动执行
- 邮件通知设置
- 新闻偏好设置

### 为什么建议尽快做

因为这两个模块可以让项目从“功能集合”变成“有用户配置系统的产品”。

## 四、AI 能力层面的升级方向

这部分是你项目最有技术含量的升级点。

---

## 1. 优化 Tool Calling 的稳定性

你现在已经接入了 tool calling，但可以继续强化。

### 建议做的优化

- 每个工具都增加更严格的参数 schema
- 对工具参数做服务端二次校验
- 对工具调用失败返回标准错误结构
- 增加工具执行耗时记录
- 把工具调用链写入日志
- 前端展示“AI 当前调用了哪个工具”

### 举例

现在如果模型传错参数，通常是后端报错然后返回失败。
更稳的做法是：

- 后端识别参数错误
- 返回“参数不合法，需要补充城市名”
- 再由模型决定是重试还是追问用户

这会让系统显得更聪明、更稳定。

---

## 2. 增加意图分类层

现在你主要依赖模型直接选择工具。
后面可以在高频业务里加一层“意图分类器”。

### 可以怎么做

先分类，再决定是否让模型自由调用：

- 天气查询
- 新闻查询
- Todo 创建
- Todo 更新
- 通用闲聊

### 好处

- 更稳定
- 更容易做统计
- 更容易排查问题
- 可以根据意图选择更便宜或更合适的模型

### 适用场景

- 高频功能
- 稳定性要求高的功能
- 高风险操作

---

## 3. 引入上下文记忆与长期记忆

现在聊天上下文主要来自单个会话历史。
未来可以增加“用户长期偏好记忆”。

### 例如

- 用户偏好用中文
- 用户通常关注 AI 和财经新闻
- 用户每天更关心 Todo 管理
- 用户偏好 OpenAI 还是 DeepSeek

### 能做什么

- 自动个性化回答
- 更精准的推荐
- 更少重复询问

### 实现思路

- 会话记忆：当前会话上下文
- 用户记忆：跨会话保存偏好
- 摘要记忆：长对话压缩后存储

---

## 4. 从“工具调用”升级到“任务规划”

现在是：

- 用户提问
- 模型调用工具
- 返回结果

未来可以升级成：

- 用户提出复杂任务
- 模型先拆步骤
- 再逐步执行
- 最后总结

例如：

`帮我根据今天的新闻和我的 Todo 安排一下明天的工作`

这就不再是单工具调用，而是组合任务：

1. 获取新闻
2. 获取 Todo
3. 分析优先级
4. 生成计划

这类能力会显著提升项目技术亮点。

---

## 5. 增加 RAG / 文档知识库能力

这是很适合后续扩展的一步。

### 可以支持

- 上传 PDF / Word / Markdown
- 让 AI 基于文件问答
- 基于项目文档生成总结
- 基于用户资料生成个性化建议

### 为什么有价值

因为这会让 AI 从“通用助手”变成“有你自己知识库的助手”。

### 适合做的场景

- 简历优化
- 学习资料问答
- 项目文档问答
- 笔记知识整理

---

## 五、工程架构层面的升级方向

这部分是让项目更像正式工程，而不是个人练手项目。

---

## 1. 做统一的 API 返回结构

现在各个接口返回格式虽然大体一致，但不完全统一。

建议统一成：

- `ok`
- `message`
- `data`
- `code`

这样做的好处：

- 前端处理更一致
- 更方便日志和错误追踪
- 以后接 SDK 或移动端更方便

---

## 2. 做统一错误处理机制

现在很多地方还是各自 `try/catch`。

可以升级成：

- 统一错误类型
- 统一错误码
- 统一日志输出
- 统一用户可见错误文案

例如区分：

- 参数错误
- 未登录
- 权限错误
- 模型调用失败
- 工具执行失败
- 第三方 API 超时

这会显著提升维护性。

---

## 3. 抽离服务层，降低 Route 复杂度

现在一部分路由已经有 service 化倾向，比如 chat-service。
可以继续推进到更多模块。

### 建议拆分

- auth service
- todo service
- news service
- profile service
- settings service
- ai orchestration service

这样每个路由只负责：

- 读参数
- 校验 session
- 调 service
- 返回 response

代码会更清晰。

---

## 4. 数据模型进一步规范化

目前整体已经有类型，但还可以加强。

### 可以优化

- DTO 和 DB Document 更清晰分层
- 表结构约束更统一
- 增加必要索引
- 增加字段级校验
- 增加软删除字段

### 例如

Todo 可增加：

- `deletedAt`
- `archivedAt`
- `order`
- `tags`

Chat 可增加：

- `summary`
- `pinned`
- `archived`
- `provider`

---

## 5. 补测试体系

这是目前大多数个人项目最缺的部分。

建议补：

- 单元测试
- API 测试
- 工具调用测试
- 关键业务流程测试

### 优先测什么

- 登录 / 注册 / 找回密码
- Todo 创建 / 更新 / 删除
- Tool call 解析逻辑
- Chat 持久化逻辑
- 新闻关键词提取

如果这部分补起来，项目可信度会提升很多。

---

## 六、安全与稳定性层面的升级方向

这一块非常重要，尤其是 AI 项目很容易忽略。

---

## 1. 强化权限与操作边界

虽然你已经做了用户级数据隔离，但还可以加强：

- 每个接口都统一鉴权
- 工具执行时再次校验用户身份
- 对高风险操作做确认
- 增加速率限制

### 特别建议

给下面这些能力加保护：

- 删除 Todo
- 删除对话
- 邮件发送
- 忘记密码验证码接口

---

## 2. 增加 Rate Limit

现在项目里很适合加限流。

### 建议加在

- 登录接口
- 注册接口
- 忘记密码发送邮件接口
- Chat 接口
- Todo AI 接口

### 作用

- 防止恶意刷接口
- 控制模型成本
- 防止邮件轰炸

---

## 3. 更完善的密码和安全策略

忘记密码流程已经有一定基础，但还可以增强：

- 验证码发送频率限制
- 单 IP 尝试次数限制
- 登录失败次数限制
- 异地登录提醒
- 更完整的会话管理

---

## 4. 第三方服务降级策略

你的项目依赖：

- OpenAI / DeepSeek / Claude
- SMTP
- RSS 新闻源
- MongoDB

所以要考虑：

- 模型不可用时是否自动切换 provider
- 新闻源失败时是否切换备用源
- SMTP 失败时是否重试
- 数据库连接异常时如何优雅报错

如果把降级策略做好，系统稳定性会高很多。

---

## 5. 增加日志和监控

这是非常值得做的一步。

建议记录：

- 用户请求
- 模型 provider
- 工具调用名称
- 工具调用参数
- 工具执行耗时
- 最终接口状态
- 错误堆栈

如果以后项目规模上来，没有这套日志系统会很难排障。

---

## 七、性能与体验层面的升级方向

---

## 1. 前端体验优化

你当前 UI 已经有一定视觉设计感，但还可以继续提升交互质量。

### 可以优化

- Skeleton 加载态
- 操作成功提示
- 失败重试按钮
- 空状态更友好
- 移动端适配
- 键盘快捷键
- 无障碍优化

### 具体建议

- 聊天输入框支持自动高度
- 消息支持复制
- Todo 支持回车、Tab 快捷操作
- 新闻模块支持一键“加入待办”

---

## 2. 性能优化

### 可以做

- 对会话列表分页
- 对消息较多的会话做虚拟滚动
- 对新闻做缓存
- 对重复查询做去重
- 对常用 RSS 源做定时抓取缓存

### 为什么重要

如果用户数据一多，当前的“全量拉取、全量渲染”策略会逐渐吃力。

---

## 3. 成本优化

AI 项目一定要考虑成本。

### 可做策略

- 闲聊走便宜模型
- 工具规划走强模型
- 长上下文做摘要压缩
- 新闻总结先缓存结果
- 高频场景减少重复请求

### 好处

- 降低 API 成本
- 提升响应速度
- 更适合长期运行

## 八、数据与运营能力升级

如果你想把这个项目从“个人项目”往“产品”方向靠，这部分很重要。

---

可以统计：

- 用户数

可以记录：

- 用户喜欢的模型
- 常查的新闻主题
- 常用语言
- 常用 Todo 分类

有了这个系统后，AI 可以做得更个性化。

## 3. 定时任务与自动化

很适合做的功能：

- 每日新闻摘要
- 每日晚间 Todo 总结
- 每周工作回顾
- 即将到期任务提醒

这类功能一上，项目会明显更像“AI 助手产品”，而不是单纯的交互页面。

## 九、你这个项目最有价值的升级路线

如果要我帮你定一个现实可执行的路线，我建议按下面顺序推进。

---

## 第一阶段：补基础短板

优先做：

- Profile / Setting 真正落地
- Todo 支持编辑和截止时间
- 聊天支持会话重命名和导出
- API 返回结构统一
- 增加基本日志
- 补关键测试

这一步完成后，项目会从“能跑”变成“结构完整”。

---

## 第二阶段：强化 AI 稳定性

优先做：

- 工具调用日志
- 工具参数二次校验
- 工具调用失败恢复
- 高风险操作确认机制
- 更细粒度的工具描述
- 增加意图分类层

这一步完成后，项目会从“AI 功能有趣”变成“AI 功能可靠”。

---

## 第三阶段：做产品差异化

优先做：

- 新闻订阅与日报
- AI 自动拆解任务
- 聊天和 Todo 联动
- 文件上传与知识库问答
- 个性化用户记忆

这一步完成后，项目会从“综合助手”开始长出自己的产品特色。

---

## 第四阶段：做成真正可扩展的平台

优先做：

- 插件式工具系统
- 统一 Agent 编排层
- 定时任务系统
